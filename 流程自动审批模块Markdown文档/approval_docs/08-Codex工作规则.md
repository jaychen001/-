# Codex 工作规则

## 1. 项目开发原则

请按照本文档约束开发流程自动审批模块。

本项目当前阶段只开发两个功能：

1. ECN 变更审批
2. 新项目下单审批

请不要一次性扩展太多复杂功能。

## 2. 开发要求

### 2.1 先设计，后编码

每次开发新功能前，请先输出：

1. 涉及的文件
2. 数据结构变化
3. 接口设计
4. 核心逻辑
5. 风险点

确认思路清楚后再写代码。

### 2.2 小步提交

每次只完成一个明确任务，例如：

- 创建数据库表
- 实现提交审批
- 实现审批通过
- 实现 ECN 新建接口
- 实现我的待审批页面

不要一次修改大量无关文件。

### 2.3 保持通用审批引擎

不要把 ECN 审批逻辑和新项目下单审批逻辑写死在审批引擎里。

审批引擎应该只关心：

- businessType
- businessId
- approvalInstance
- approvalNode
- approvalLog

业务差异应该放在业务模块中处理。

### 2.4 状态必须统一

业务单据状态和审批实例状态需要保持一致。

所有状态统一使用：

```text
draft
pending
approved
rejected
returned
withdrawn
```

审批节点状态统一使用：

```text
waiting
pending
approved
rejected
skipped
```

不要创建含义重复的新状态。

### 2.5 所有审批动作必须记录日志

以下动作必须写入 approval_logs：

- create
- submit
- approve
- reject
- return
- withdraw
- auto_next
- complete
- cc

日志至少包含：

- instanceId
- nodeId
- businessType
- businessId
- operatorId
- action
- comment
- fromStatus
- toStatus
- createdAt

### 2.6 审批流转必须使用事务

以下操作必须在数据库事务中完成：

- 提交审批
- 审批通过
- 审批驳回
- 退回修改
- 撤回审批

避免出现业务单据状态已经改变，但审批节点或日志未更新的问题。

## 3. 第一阶段不要做的功能

当前阶段不要开发：

- 并行审批
- 会签
- 或签
- 条件审批
- 流程设计器
- 企业微信通知
- 移动端审批
- 复杂动态审批人
- AI 自动判断是否通过

这些功能后续再扩展。

## 4. 推荐开发顺序

请严格按以下顺序开发：

1. 创建数据库表结构
2. 初始化角色和审批模板
3. 实现通用审批引擎
4. 实现 ECN 变更申请
5. 实现新项目下单申请
6. 实现审批中心
7. 实现审批详情页
8. 实现审批日志
9. 实现抄送功能
10. 实现权限控制
11. 编写测试用例

## 5. 每次输出代码前需要说明

每次开始写代码前，请先说明：

```text
本次准备修改哪些文件？
本次实现哪个功能？
本次会不会影响已有功能？
本次完成后的验证方式是什么？
```

## 6. 代码质量要求

1. 代码结构清晰。
2. 命名统一。
3. 不写重复逻辑。
4. 通用审批逻辑不要写在具体业务模块里。
5. 业务模块可以调用审批引擎。
6. 所有异常需要返回明确错误码。
7. 所有接口需要基础参数校验。
8. 所有审批操作必须校验权限。
9. 所有审批状态变化必须写日志。
10. 重要操作需要加事务。

## 7. 命名规范

### 7.1 业务类型

```text
ECN_APPROVAL
PROJECT_ORDER_APPROVAL
```

### 7.2 角色编码

```text
PROJECT_MANAGER
RND_MANAGER
GENERAL_MANAGER
```

### 7.3 状态编码

审批主状态：

```text
draft
pending
approved
rejected
returned
withdrawn
```

审批节点状态：

```text
waiting
pending
approved
rejected
skipped
```

## 8. 开发完成后的检查

每个功能完成后，请检查：

- 是否能正常创建数据
- 是否能正常提交审批
- 是否能自动生成审批节点
- 是否能正确判断当前审批人
- 是否能审批通过
- 是否能驳回
- 是否能退回修改
- 是否能写入审批日志
- 是否能同步业务单据状态
- 是否有权限漏洞
- 是否有事务问题

